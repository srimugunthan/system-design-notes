# L4: GPU Architecture & Roofline

## Introduction

GPUs are the workhorses of LLM inference due to their parallel processing capabilities. Understanding GPU architecture and the Roofline model is essential for optimizing inference performance. This section covers streaming multiprocessors, tensor cores, and how to analyze compute vs. memory bottlenecks.

## GPU Architecture Overview

Modern GPUs like NVIDIA's Ampere or Hopper architectures are designed for massive parallelism.

### Streaming Multiprocessors (SMs)

SMs are the basic processing units containing CUDA cores, tensor cores, and shared memory.

#### Theoretical Background

Each SM can execute thousands of threads concurrently. For inference, SMs handle matrix multiplications and attention computations.

#### Practical Implementation

In CUDA, kernels are launched with thread blocks assigned to SMs.

ASCII Diagram of GPU Architecture:

```
GPU Chip
+-------------------+
| SM1 SM2 SM3 SM4   |
|                   |
| Memory Hierarchy  |
| L1 L2 HBM         |
+-------------------+
```

- NVIDIA CUDA Programming Guide - https://docs.nvidia.com/cuda/cuda-c-programming-guide/

## Tensor Cores

Tensor cores accelerate matrix multiplications, crucial for transformer computations.

### Theoretical Background

Tensor cores perform 4x4 matrix multiplications in one clock cycle, optimized for FP16/BF16 operations.

For LLM inference, tensor cores handle GEMM operations in attention and feed-forward layers.

### Practical Implementation

In PyTorch, use `torch.nn.functional.linear` with appropriate dtypes to leverage tensor cores.

For quantized models, INT8/INT4 tensor core operations are used.

## Roofline Model

The Roofline model analyzes performance bottlenecks by plotting operational intensity vs. performance.

### Theoretical Background

Roofline: Performance = min(Peak Compute, Peak Memory Bandwidth * Operational Intensity)

Operational Intensity = FLOPs / Bytes transferred

#### Compute Bound

When OI > OI_threshold, performance limited by compute capability.

#### Memory Bound

When OI < OI_threshold, performance limited by memory bandwidth.

### Practical Implementation

For LLM inference:
- Attention: Often memory-bound due to KV cache accesses
- Feed-forward: Compute-bound with large matrices

ASCII Diagram of Roofline:

```
Performance
    ^
    |     /\
    |    /  \  Compute Bound
    |   /    \
    |  /      \
    | /        \
    |/          \ Memory Bound
    +-------------> Operational Intensity
   OI_threshold
```

To optimize:
- Fuse kernels to increase OI
- Use FlashAttention to reduce memory accesses

Reference:
- Paper: Williams, S., et al. (2009). Roofline: An Insightful Visual Performance Model for Multicore Architectures. CACM
- Blog: Roofline Analysis for Deep Learning - https://www.nvidia.com/en-us/on-demand/session/gtcspring21-s31633/

## Compute vs. Memory Bound in LLM Inference

### Memory-Bound Operations

- KV cache reads/writes
- Embedding lookups
- Long-sequence attention

### Compute-Bound Operations

- Large matrix multiplications in FFN
- High-precision computations

### Optimization Strategies

- Quantization to reduce memory bandwidth needs
- Kernel fusion to improve OI
- PagedAttention for efficient memory management

Reference:
- Blog: Optimizing Transformer Inference - https://huggingface.co/blog/optimizing-transformer-inference

## Conclusion

Understanding GPU architecture and Roofline analysis allows inference engineers to identify and address performance bottlenecks effectively. Balancing compute and memory optimizations is key to high-throughput LLM serving.
