# L12: Production Systems

## Introduction

Deploying LLMs in production requires careful consideration of reliability, scalability, and safety. Cold starts, deployment strategies, caching, and guardrails ensure robust serving. This section covers batch vs online inference and production best practices.

## Cold Starts

Latency spike when loading models.

### Theoretical Background

Model loading from disk/network to GPU memory.

Affects first request after idle period.

### Practical Implementation

- Pre-warm instances
- Model caching in memory
- Lazy loading for multi-model serving

Reference:
- Blog: Cold Starts in Serverless ML - https://aws.amazon.com/blogs/compute/optimizing-aws-lambda-cold-starts/

## Deployment Strategies

### Canary Deployments

Gradual rollout of new model versions.

#### Theoretical Background

Route percentage of traffic to new version, monitor metrics.

Rollback if issues detected.

#### Practical Implementation

In Kubernetes: Istio traffic splitting.

For LLMs: A/B test on quality metrics.

ASCII Diagram:

```
Traffic
      |
      +-------------------+
      | 90% Old Model     |
      | 10% New Model     |
      +-------------------+
      |
      v
Monitor Metrics
      |
      +-------------------+
      | Quality OK?       |
      +-------------------+
      | Yes: Increase     |
      | No: Rollback      |
      +-------------------+
```

Reference:
- Blog: Canary Deployments for ML - https://neptune.ai/blog/canary-deployment-ml-models

Route requests based on cache state.

### Theoretical Background

For KV cache sharing, route to instances with relevant cached prefixes.

### Practical Implementation

In vLLM: session-based routing for conversation continuity.

Ensure safe, appropriate outputs.

### Theoretical Background

Content moderation, fact-checking, toxicity filtering.

### Practical Implementation

## Batch vs Online Inference

### Online Inference

Real-time, low-latency responses.

#### Theoretical Background

Request-response pattern, sub-second latency.

#### Practical Implementation

- GPU instances for parallel processing
- Auto-scaling based on load

### Batch Inference

Process multiple requests together.

#### Theoretical Background

Higher throughput, higher latency.

For offline tasks: data processing, evaluation.

#### Practical Implementation

- Queue requests
- Process in large batches
- Cost-effective for non-real-time

ASCII Diagram:

```
Online: Request -> Process -> Response (fast)
Batch:  Queue -> Batch -> Process -> Responses (efficient)
```

## Production Best Practices

### Monitoring

- Latency, throughput, error rates
- Model quality metrics
- Resource utilization

### Scaling

- Horizontal scaling with load balancers
- Vertical scaling for larger models
- Multi-region deployment

### Reliability

- Health checks
- Circuit breakers
- Graceful degradation

## Conclusion

Production LLM systems require balancing performance, reliability, and safety. Deployment strategies like canary releases, caching optimizations, and guardrails ensure smooth operation. Choosing between batch and online inference depends on latency requirements and use cases.
